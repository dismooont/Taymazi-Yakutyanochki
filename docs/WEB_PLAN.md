# План: веб-интерфейс для CLIP zero-shot поиска

Документ описывает, как надстроить над существующим ядром (CLIP + FAISS) многопользовательский
веб-интерфейс. Telegram-бот остаётся и продолжает работать — он переиспользует то же ядро.

Принятые решения:

| Вопрос | Решение |
|---|---|
| Идентификация | **Логин + пароль (основной способ)**, Telegram Login Widget — дополнительный |
| Фронтенд | React + Vite (SPA), бэкенд — API-only FastAPI |
| Долгие операции | Фоновые задачи с прогрессом (job_id + опрос статуса) |

Аккаунт не привязан к способу входа: `users` отделены от `identities`, и к одному аккаунту
можно привязать и пароль, и Telegram. Telegram полностью отключается флагом
`TELEGRAM_AUTH_ENABLED=0` — приложение остаётся работоспособным.

---

## 1. Что есть сейчас и чего не хватает

Текущее состояние (`src/clip_zero_shot_search.py`, `bot/inference.py`):

* Индекс **один глобальный** — папка `index/`, `IndexFlatIP` + `images_meta.json` (список,
  позиция в котором = номер вектора в FAISS).
* `SearchEngine` умеет `search_by_text`, `search_by_image`, `add_image` (по одной картинке).
* Модель CLIP (~600 МБ RAM) грузится один раз на процесс.
* Пути в `images_meta.json` — относительные и в стиле ОС, где строился индекс
  (есть костыль `_portable_path`).

Чего нет и что придётся написать:

1. **Изоляция баз по пользователям.** Сейчас `index_dir` зашит один на всё приложение.
2. **Создание пустой базы.** `SearchEngine.__init__` падает, если `images.index` не существует.
3. **Удаление фото.** Не реализовано вообще — а это пункт (3) сценария.
4. **Учёт объёма** (кол-во фото + байты) — нигде не считается.
5. **Импорт zip / экспорт zip.**
6. **Пакетное добавление с прогрессом** — сейчас `add_image` строго по одной картинке.

Пункты 2–4 требуют рефакторинга ядра, а не только новых ручек API. Это самая рискованная часть
работы, поэтому она идёт первым этапом.

## 2. Архитектура процессов

```
                    ┌──────────────────────────────────────────┐
   браузер  ──────▶ │ FastAPI (uvicorn, ОДИН воркер)           │
   (React SPA)      │                                          │
                    │  ├── REST API + сессии (cookie)          │
                    │  ├── ModelHolder: CLIP загружен 1 раз    │
                    │  ├── StoreCache: LRU открытых баз (3–5)  │
                    │  └── JobQueue: asyncio.Queue + 1 воркер  │
                    │        └── asyncio.to_thread(...)        │
                    └──────────────┬───────────────────────────┘
                                   │
              ┌────────────────────┴────────────────────┐
              │ data/app.db (SQLite)   data/users/...   │
              └─────────────────────────────────────────┘

   Telegram-бот — отдельный контейнер, тот же пакет core/, свой индекс.
```

**Почему один процесс, а не web + отдельный worker + Redis.** Эмбеддинги считает CLIP, а он
занимает ~600 МБ RAM. Отдельный воркер-процесс означает вторую копию модели в памяти. Поэтому
модель, API и очередь живут в одном процессе; тяжёлый инференс уходит в `asyncio.to_thread`
(ровно тот приём, что уже применён в `bot/bot.py`), чтобы не блокировать event loop.

Следствия, о которых надо помнить:

* uvicorn запускать строго с `--workers 1`. Несколько воркеров = несколько копий модели
  и гонки при записи в один и тот же `images.index`.
* Очередь задач — одна на всё приложение, последовательная. При нескольких активных
  пользователях задачи выстраиваются в очередь; UI должен показывать позицию («3-й в очереди»).
* На каждую базу — свой `asyncio.Lock`. Поиск по базе и запись в неё не должны пересекаться.

## 3. Хранение

### 3.1. Файлы

```
data/
├── app.db                              # SQLite: пользователи, базы, задачи
└── users/
    └── <user_id>/                       # внутренний uuid, НЕ telegram id
        └── databases/
            └── <db_id>/                # uuid4
                ├── images.index        # FAISS IndexFlatIP
                ├── images_meta.json    # [{photo_id, filename, bytes, added_at}]
                ├── captions.index      # опционально (см. 3.3)
                ├── captions_meta.json
                ├── translate_cache.json
                ├── photos/             # оригиналы, имя = <photo_id>.jpg
                ├── thumbs/             # превью 320px (WebP) для галереи и выдачи
                └── tmp/                # распаковка архивов, сборка экспорта
```

Внутри базы в meta хранится **только имя файла**, а не путь — корень базы известен из БД.
Это убирает текущую проблему с Windows-путями в `images_meta.json` и делает базу переносимой.

### 3.2. SQLite (`data/app.db`, режим WAL)

```sql
users(id TEXT PK,                        -- внутренний uuid4, не зависит от способа входа
      login TEXT UNIQUE,                 -- логин, хранится в нижнем регистре
      display_name TEXT,
      password_hash TEXT,                -- argon2id; NULL, если вход только через Telegram
      email TEXT,                        -- опционально, для восстановления пароля
      created_at, last_seen_at)

identities(provider TEXT,                -- 'telegram' (для пароля отдельная строка не нужна)
           provider_user_id TEXT,
           user_id TEXT FK,
           linked_at,
           PRIMARY KEY (provider, provider_user_id))

sessions(id TEXT PK,                     -- случайный токен, в cookie кладётся он
         user_id TEXT FK,
         created_at, expires_at, last_seen_at,
         user_agent TEXT, ip TEXT)

databases(id TEXT PK, user_id TEXT FK, name TEXT,
          photos_count INT DEFAULT 0,      -- счётчики, обновляются транзакционно
          photos_bytes INT DEFAULT 0,
          index_bytes  INT DEFAULT 0,
          has_captions INT DEFAULT 0,
          status TEXT,                     -- ready | indexing | error
          created_at, updated_at)

jobs(id TEXT PK, user_id TEXT, database_id TEXT,
     kind TEXT,                            -- import_zip | add_photos | export_zip | compact
     status TEXT,                          -- queued | running | done | error
     progress_done INT, progress_total INT,
     message TEXT, result_path TEXT,
     created_at, started_at, finished_at)
```

Сессии теперь **серверные** (таблица `sessions`), а не просто подписанная cookie. С паролями
это принципиально: нужна возможность разлогинить все устройства при смене пароля и отозвать
украденную сессию. В cookie кладётся только случайный токен сессии.

Счётчики держим в БД, а не пересчитываем обходом папки — иначе «текущий объём базы» на каждый
рендер будет стоить `os.stat()` по тысячам файлов. Раз в сутки (или по запросу) — сверка
с реальностью и починка расхождений.

### 3.3. Про индекс подписей

`captions.index` в текущем проекте строится из `captions.csv` (COCO). У пользовательской базы
подписей нет, поэтому:

* поиск **текст → фото** работает всегда (текст эмбеддится на лету — CSV не нужен);
* поиск **фото → подписи** для пользовательских баз недоступен.

В UI при поиске по картинке для базы без подписей блок «релевантные подписи» просто не
показывается (`has_captions = 0`). Альтернатива на будущее — автогенерация подписей моделью
типа BLIP, но это отдельная задача и в этот план не входит.

## 4. Рефакторинг ядра (`core/`)

Выделяем пакет `core/`, из которого одинаково импортируют и веб, и бот:

```
core/
├── model.py        # ModelHolder: CLIP + processor, singleton, потокобезопасный
├── embeddings.py   # compute_image_embeddings / compute_text_embeddings (переезд из src/)
├── translate.py    # translate_ru_to_en + кэш (переезд из src/)
└── store.py        # IndexStore — вся работа с одной базой
```

`src/clip_zero_shot_search.py` остаётся как CLI-обёртка над `core/` (обратная совместимость
команд `build/add/text/image` и скрипта `scripts/eval_recall.py` не ломается).

### 4.1. `IndexStore` — API

```python
class IndexStore:
    @classmethod
    def create_empty(cls, path: Path) -> "IndexStore"      # пустая база
    @classmethod
    def open(cls, path: Path) -> "IndexStore"

    def add_photos(self, files: list[Path], on_progress=None) -> AddResult
    def delete_photos(self, photo_ids: list[str]) -> int
    def search_text(self, query: str, top_k: int, translate: bool) -> SearchResult
    def search_image(self, image: Image.Image, top_k: int) -> SearchResult
    def stats(self) -> Stats            # photos_count, photos_bytes, index_bytes
    def iter_photo_files(self)          # для экспорта
```

Три момента, которые надо сделать аккуратно:

**Пустая база.** `IndexFlatIP(dim)` без единого вектора — валиден, `search` по нему вернёт
`-1` в индексах, это надо отфильтровать (сейчас код упадёт по `meta[-1]`). Размерность брать
из `model.config.projection_dim`, а не хардкодить 512.

**Удаление.** `IndexFlatIP` хранит сырые векторы, поэтому пересборка базы после удаления
не требует пересчёта эмбеддингов: `index.reconstruct_n(0, index.ntotal)` возвращает всю
матрицу, выкидываем строки удалённых, собираем новый `IndexFlatIP`. Для 5000 векторов это
матрица 5000×512 float32 ≈ 10 МБ — операция занимает миллисекунды. Порядок действий:

1. под локом базы: reconstruct → фильтрация строк → новый индекс → `write_index` во временный
   файл → атомарный `os.replace`;
2. тем же образом чистим `captions.index` от строк с этим `image_id` (если он есть);
3. удаляем файлы `photos/<id>.jpg` и `thumbs/<id>.webp`;
4. в одной транзакции уменьшаем счётчики в БД.

Если база вырастет до сотен тысяч фото, пересборка на каждое удаление станет заметной — тогда
переходим на мягкое удаление (флаг в meta + фильтрация выдачи с запасом по `top_k`) и
компактизацию по порогу. Для целевых объёмов проекта это преждевременно.

**Пакетное добавление.** `add_photos` считает эмбеддинги батчами по 32 (как `build`), а не по
одной картинке, и дёргает `on_progress(done, total)` после каждого батча. На CPU это разница
примерно в 3–5 раз по сравнению с поштучным вызовом.

### 4.2. `StoreCache`

LRU на 3–5 открытых `IndexStore` + выгрузка по TTL (например, 15 минут без обращений).
Держать в памяти все базы всех пользователей нельзя: `images.index` на 5000 фото — это 10 МБ,
но `captions.index` того же датасета — уже 50 МБ.

## 5. API

Все ручки под `/api`, JSON. Сессия — HttpOnly-cookie с токеном из таблицы `sessions`,
`SameSite=Lax`, `Secure` в проде.

### Авторизация
| Метод | Путь | Описание |
|---|---|---|
| `POST` | `/api/auth/register` | `{login, password, display_name?, email?}` → создаёт аккаунт и сессию |
| `POST` | `/api/auth/login` | `{login, password}` → сессия |
| `POST` | `/api/auth/logout` | завершить текущую сессию |
| `POST` | `/api/auth/logout-all` | завершить все сессии пользователя |
| `GET` | `/api/me` | текущий пользователь или 401 |
| `POST` | `/api/me/password` | `{old_password, new_password}` — смена пароля |
| `POST` | `/api/auth/telegram` | опционально: payload Login Widget → вход или привязка к текущему аккаунту |
| `DELETE` | `/api/me/identities/telegram` | отвязать Telegram |

Логика связывания: если пользователь вошёл по паролю и затем прислал валидный payload
Telegram — создаётся строка в `identities` для его `user_id`. Если payload пришёл без активной
сессии и такой `identities` уже есть — обычный вход. Если нет — создаётся новый аккаунт
с `password_hash = NULL` и сгенерированным логином; пароль он сможет задать позже
(`POST /api/me/password` с пустым `old_password`, если хеша ещё нет).

Проверка виджета остаётся прежней: HMAC-SHA256 от `data_check_string` с ключом
`SHA256(bot_token)`, сверка с полем `hash`, отказ при `auth_date` старше 24 часов.

**Восстановление пароля.** Полноценный сброс через письмо требует SMTP и подтверждения адреса —
это отдельная инфраструктура, в объём проекта не входит. Поэтому в v1: `email` необязателен и
хранится справочно, а варианта восстановления два — вход через привязанный Telegram (если
привязан) либо ручной сброс администратором через CLI-команду. Это надо честно написать
в форме регистрации, а не делать вид, что «забыли пароль» работает.

### Базы
| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/api/databases` | список баз пользователя со статистикой |
| `POST` | `/api/databases` | создать пустую базу `{name}` |
| `PATCH` | `/api/databases/{id}` | переименовать |
| `DELETE` | `/api/databases/{id}` | удалить базу целиком |
| `POST` | `/api/databases/{id}/import` | multipart zip → `{job_id}` |
| `GET` | `/api/databases/{id}/stats` | `{photos_count, photos_bytes, index_bytes}` |

Создание базы из архива = `POST /api/databases` + `POST /api/databases/{id}/import`.
Так «загрузить архив» и «создать пустую» — один и тот же путь, отличающийся вторым шагом.

### Фото
| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/api/databases/{id}/photos?offset&limit` | галерея, постранично |
| `POST` | `/api/databases/{id}/photos` | multipart, N файлов → `{job_id}` (при ≤3 файлах — синхронно) |
| `DELETE` | `/api/databases/{id}/photos/{photo_id}` | удалить одно фото |
| `POST` | `/api/databases/{id}/photos/delete` | удалить пачкой `{photo_ids: []}` |
| `GET` | `/api/databases/{id}/photos/{photo_id}/thumb` | превью |
| `GET` | `/api/databases/{id}/photos/{photo_id}/file` | оригинал |

Картинки отдаются **только через API с проверкой владельца**, не статикой из `data/users/`.
Иначе чужую базу можно перебрать по URL.

### Поиск
| Метод | Путь | Описание |
|---|---|---|
| `POST` | `/api/databases/{id}/search/text` | `{query, top_k, translate}` → `{used_query, results[]}` |
| `POST` | `/api/databases/{id}/search/image` | multipart файл → `{results[], captions[]}` |

В ответ на текстовый поиск возвращаем и `used_query` (переведённую строку) — в боте это уже
показывается пользователю, на вебе полезно так же: видно, во что превратился русский запрос.

### Экспорт
| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/api/databases/{id}/export.zip` | потоковая отдача zip |

Фото — это JPEG, они уже сжаты, поэтому zip собирается с `ZIP_STORED` и стримится на лету
(`StreamingResponse` + генератор), без сборки временного файла на диске. Отдельная фоновая
задача под экспорт не нужна. В архив кладём `photos/` и `manifest.json` (id, имя, дата,
размер) — чтобы такой архив можно было залить обратно.

### Задачи
| Метод | Путь | Описание |
|---|---|---|
| `GET` | `/api/jobs/{id}` | `{status, progress:{done,total}, queue_position, message}` |
| `POST` | `/api/jobs/{id}/cancel` | отмена (кооперативная, по флагу между батчами) |

Опрос раз в секунду, пока `status in (queued, running)`. SSE — необязательное улучшение;
на одну активную задачу пользователя поллинг проще и надёжнее.

## 6. Фоновые задачи

Очередь: `asyncio.Queue` + один воркер-таск, стартующий в `lifespan` приложения.

Типы задач:

* **`import_zip`** — распаковать архив, отфильтровать картинки, посчитать эмбеддинги батчами,
  сделать превью, записать индекс. Прогресс по количеству обработанных файлов.
* **`add_photos`** — то же без распаковки.
* **`compact`** — обслуживание, вручную/по расписанию.

Индекс пишется на диск **не после каждого батча, а раз в ~500 фото и в конце** — `write_index`
на 5000 векторов это 10 МБ на диск, делать это 150 раз подряд бессмысленно. При падении
процесса теряется незаписанный хвост; на старте приложения базы со `status = indexing`
помечаются как требующие сверки meta ↔ ntotal.

Задачи переживают перезапуск только как «оборванные»: при старте все `running` переводятся
в `error` с понятным сообщением. Полноценный restart-resume для этого проекта не нужен.

## 7. Безопасность

### 7.1. Пароли и сессии

Раз мы храним пароли, ответственность за них теперь наша. Минимум:

* **Хеширование — argon2id** (`argon2-cffi`), параметры по умолчанию из библиотеки.
  Не MD5/SHA — они рассчитаны на скорость, то есть на удобство перебора. bcrypt приемлем
  как запасной вариант, но помнит только первые 72 байта пароля.
* **Проверка пароля должна занимать одинаковое время** независимо от того, существует логин
  или нет: при отсутствии пользователя всё равно прогонять argon2 по фиктивному хешу.
  Иначе по времени ответа перебираются существующие логины.
* **Одинаковый текст ошибки** для «нет такого логина» и «неверный пароль» — «Неверный логин
  или пароль». При регистрации занятый логин скрыть не получится, поэтому там просто
  ограничиваем частоту.
* **Rate limit на вход**: скользящее окно по логину (5 попыток / 15 мин → временная блокировка
  с экспоненциальной задержкой) **и** по IP. Только по IP недостаточно — ботнет обойдёт;
  только по логину — можно намеренно заблокировать чужой аккаунт, поэтому блокировка временная,
  а не до вмешательства администратора.
* **Ротация сессии** при входе и смене пароля: `POST /api/me/password` удаляет все сессии
  пользователя, кроме текущей.
* **Требования к паролю**: минимум 10 символов, проверка по списку самых частых паролей.
  Навязывать «спецсимвол + цифра + заглавная» не нужно — это ухудшает пароли, а не улучшает.
* **Логин**: 3–32 символа, `[a-z0-9_.-]`, нормализация в нижний регистр перед сравнением
  и записью, иначе появятся `Ivan` и `ivan` как разные аккаунты.
* **CSRF**: `SameSite=Lax` закрывает основной вектор, но так как API принимает JSON, добавляем
  обязательный заголовок `X-Requested-With` на все мутирующие запросы — простую HTML-форму
  с чужого сайта так не подделать.
* **Пароль уходит по сети открытым текстом внутри TLS** — HTTPS в проде обязателен
  (для Telegram-виджета он был нужен и раньше, теперь это требование и для входа по паролю).
* **Не логировать** тело запросов `/api/auth/*` — иначе пароли окажутся в логах uvicorn.

### 7.2. Загрузка архивов

Приём zip-архива от пользователя — самая опасная точка. Обязательный чек-лист:

* **Zip-slip**: игнорировать абсолютные пути и `..` в именах членов архива; из каждого имени
  брать только `basename`. Никогда не доверять `ZipFile.extractall` на пользовательских данных.
* **Zip-бомба**: до распаковки просуммировать `ZipInfo.file_size` и отказать при превышении
  лимита; ограничить количество членов архива и размер одного файла.
* **Проверка содержимого**: расширение из белого списка (`.jpg/.jpeg/.png/.webp`) **и**
  успешный `PIL.Image.open(...).verify()`. Расширение само по себе ничего не гарантирует.
* **Symlink-члены архива** — пропускать.
* **Квоты на пользователя**: например, 5 баз, 5000 фото и 3 ГБ суммарно. Проверять до начала
  задачи и повторно по ходу.
* **Лимит размера запроса** на уровне nginx (`client_max_body_size`) и приложения.
* **Проверка владельца** в каждой ручке, где есть `{database_id}` — единая зависимость FastAPI
  `get_owned_database(db_id, user)`, чтобы её нельзя было забыть.
* **Rate limit** на поиск (CPU-bound, один процесс) — например, 10 запросов/мин на пользователя.
* **EXIF-ориентация** при создании превью (`ImageOps.exif_transpose`), иначе часть фото
  в галерее будет лежать на боку.
* Cookie: `HttpOnly`, `SameSite=Lax`, `Secure`, срок жизни ~30 дней.

### Экраны по пользовательскому сценарию

**1. Login / Register** — две вкладки на одной странице:

```
        ( Вход | Регистрация )

        Логин     [________________]
        Пароль    [________________]  👁
                  [      Войти      ]

        ─────────  или  ─────────
        [ Войти через Telegram ]      ← скрыт, если TELEGRAM_AUTH_ENABLED=0
```

На вкладке регистрации — те же поля плюс подтверждение пароля и индикатор его силы.
Валидация логина/длины пароля дублируется на фронте (быстрая обратная связь) и на бэке
(единственная, которой можно верить). Под формой — честная строка о том, что
самостоятельное восстановление пароля не предусмотрено, стоит привязать Telegram
в настройках профиля.

**2. Databases** (пункты сценария «выбор» и «(5) вернуться к предыдущим базам»):

```
┌─────────────────────────────┬─────────────────────────────┐
│  📦 Загрузить архив с фото  │  ✨ Создать новую базу       │
│  перетащите .zip сюда       │  пустая, добавите фото потом │
└─────────────────────────────┴─────────────────────────────┘

Мои базы
┌──────────────────────────────────────────────────────────┐
│ Отпуск 2024        1 240 фото · 2,1 ГБ    12.07  [→][⋯]  │
│ Работа               87 фото · 154 МБ     03.07  [→][⋯]  │
│ Индексируется…      ▓▓▓▓▓▓░░░░ 620/1240   [отменить]     │
└──────────────────────────────────────────────────────────┘
```

**3. Workspace** `/db/:id` — всё остальное:

```
[← Мои базы]   Отпуск 2024   ●  1 240 фото · 2,1 ГБ    [＋ Добавить] [⬇ Экспорт]

 ( Поиск по тексту | Поиск по фото )
 [ рыжий кот на подоконнике                              ] [Найти]
 → запрос переведён: "a ginger cat on a windowsill"

 [результаты: сетка карточек, score, клик — просмотр, ⋯ — удалить]

 ( Результаты | Вся база )   ← вкладка «Вся база»: галерея с пагинацией и удалением
```

Объём базы (`N фото · X МБ/ГБ`) виден в шапке Workspace и в каждой строке списка баз —
это требование сценария. Форматирование байт на фронте; бэкенд отдаёт сырые числа.

### Мелочи, которые сильно влияют на ощущение от интерфейса

* Поиск по фото — тот же дропзон, что и загрузка: перетащил картинку → сразу выдача.
* Удаление фото — оптимистичное, с «Отменить» в тосте 5 секунд, реальный запрос по истечении.
* Пустая база не должна выглядеть сломанной: вместо пустой выдачи — приглашение добавить фото.
* Долгая задача не блокирует: можно уйти на список баз, прогресс продолжает тикать в строке базы.
* Скелетоны вместо спиннеров в галерее — превью 320px грузятся заметно.

## 9. Деплой

В `docker-compose.yml` добавляется сервис `web` рядом с существующим `bot`:

```yaml
web:
  build: { context: ., dockerfile: Dockerfile.web }
  env_file: .env
  volumes:
    - ./data:/app/data          # базы пользователей (rw!)
  ports: ["8000:8000"]
  restart: unless-stopped
```

* `Dockerfile.web` — базируется на том же слое с CPU-only torch, что и текущий `Dockerfile`
  (сборка весов CLIP кэшируется в образе или в volume `~/.cache/huggingface`).
* Фронт собирается в статику (`npm run build`) и отдаётся nginx; nginx же проксирует `/api`.
* Вход по логину/паролю работает на `localhost` без всяких условий — костыль `DEV_FAKE_USER`
  из первой редакции плана больше не нужен, разработчик просто регистрирует себе аккаунт.
* **Telegram Login Widget по-прежнему требует публичный HTTPS-домен**, привязанный к боту
  через `/setdomain` у @BotFather. Поэтому локально его держим выключенным
  (`TELEGRAM_AUTH_ENABLED=0`) — на вход по паролю это никак не влияет.
* В проде HTTPS обязателен уже не только из-за виджета, а потому что по сети идут пароли:
  сертификат от Let's Encrypt, редирект с http, заголовок HSTS.

Новые переменные `.env`: `PUBLIC_URL`, `DATA_DIR`, `TELEGRAM_AUTH_ENABLED`,
`MAX_DB_PER_USER`, `MAX_PHOTOS_PER_DB`, `MAX_BYTES_PER_USER`,
`SESSION_TTL_DAYS`, `REGISTRATION_OPEN` (позволяет закрыть регистрацию после набора
нужных пользователей — для учебного стенда полезно).

Новые зависимости бэкенда: `fastapi`, `uvicorn[standard]`, `python-multipart`,
`argon2-cffi`, `itsdangerous` (подпись cookie), `pillow` (уже есть).

Административная CLI-команда для сброса пароля (раз уж самообслуживания нет):

```bash
python -m web.admin reset-password --login ivan --password <новый>
python -m web.admin list-users
```

## 11. Риски

| Риск | Смягчение |
|---|---|
| RAM: модель + несколько открытых баз | LRU на 3–5 баз, выгрузка по TTL, мониторинг RSS |
| Один воркер — очередь при нескольких пользователях | показывать позицию в очереди; при росте нагрузки выносить воркер в отдельный процесс (ценой второй копии модели) |
| Telegram-виджет не работает на localhost | локально `TELEGRAM_AUTH_ENABLED=0`, вход по паролю от него не зависит |
| Хранение паролей — новая зона ответственности | argon2id, rate limit на вход, никаких паролей в логах, HTTPS обязателен (см. 7.1) |
| Нет самостоятельного восстановления пароля | честно сказано в UI; запасные пути — привязанный Telegram и CLI-сброс администратором |
| Открытая регистрация на публичном стенде = чужие люди жгут ваш CPU | `REGISTRATION_OPEN=0` после набора пользователей, квоты, rate limit на поиск |
| Индексация 5000 фото на CPU — десятки минут | честный прогресс + возможность уйти со страницы; в README указать ориентир по времени |
| Расхождение `images_meta.json` и `index.ntotal` после сбоя | атомарная запись через временный файл + `os.replace`; сверка при открытии базы |
| Поиск «фото → подписи» недоступен для пользовательских баз | скрывать блок в UI при `has_captions = 0`, объяснить в README |
| Пользователь заливает архив на 10 ГБ | квоты + проверка распакованного размера до начала работы |
