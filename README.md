# CLIP Zero-Shot Image ⇄ Text Search

Прототип семантического поиска "картинка по тексту" и "текст/похожие картинки по картинке"
на базе CLIP (`openai/clip-vit-base-patch32`) и FAISS, на датасете MS COCO. Без дообучения —
используются готовые предобученные веса CLIP (zero-shot).

Поддерживается перевод текстовых запросов с русского на английский (CLIP обучен на английских
подписях, поэтому русский запрос сначала переводится). Переводы кэшируются в
`index/translate_cache.json`, чтобы не тратить лимиты внешнего API на повторные запросы.

Система поддерживает **инкрементальное добавление новых изображений** в уже построенный индекс —
без пересчёта эмбеддингов для всего датасета заново. Индекс растёт по мере поступления новых
изображений, а не пересобирается с нуля каждый раз.

## Интерфейсы

Пользоваться поиском можно тремя способами, и все они работают с одними и теми же данными:

| Интерфейс | Для чего | Раздел |
|---|---|---|
| **Веб-приложение** | у каждого пользователя своя база: загрузка архива, поиск, галерея, выгрузка zip | 9 |
| **Telegram-бот** | своя база у каждого чата: бот запоминает присланные в чат снимки и ищет по ним | 9.4 |
| **CLI** | построение индекса, оценка качества, разовые запросы | 6 |

Ключевое архитектурное свойство: **модель CLIP загружена ровно в одном процессе** — в API
веб-приложения. Бот к ней не обращается напрямую, а ходит в тот же API по HTTP, поэтому веса
(~600 МБ) существуют в единственном экземпляре, а квоты и правила доступа одинаковы везде.

**Важно про Python 3.13**: на момент написания README некоторые версии `torch`/`faiss-cpu`

с установкой зависимостей — используйте Python 3.10 или 3.11.

**Про размер данных**: полное подмножество val2017 MS COCO — это ~5000 изображений (~1 ГБ архив
изображений + ~240 МБ аннотаций). Для быстрых экспериментов можно ограничиться меньшим
подмножеством (см. раздел 5). Текущий рабочий индекс построен на полном val2017:
**5000 изображений, 25 014 подписей** (плюс фото, добавленные через бота инкрементально).

## 3. Установка (локально, вне Colab)

### 3.1. Клонируйте/распакуйте проект и создайте виртуальное окружение

```bash
cd clip_zero_shot_search
python3 -m venv .venv

# Linux / macOS
source .venv/bin/activate
# Windows (PowerShell)
.venv\Scripts\Activate.ps1
```

### 3.2. Установите зависимости

```bash
pip install -r requirements.txt          # CLI и ядро
pip install -r requirements-web.txt      # + веб-приложение (раздел 9)
pip install -r requirements-dev.txt      # + pytest, если будете гонять тесты
```

Зависимости бота ставятся отдельно и намеренно лёгкие — ни torch, ни faiss там нет:

```bash
pip install -r bot/requirements.txt
```

Если у вас NVIDIA GPU и нужна GPU-версия PyTorch — сначала поставьте `torch` отдельно
по инструкции с [pytorch.org](https://pytorch.org/get-started/locally/) под вашу версию CUDA,
и только потом остальные зависимости из `requirements.txt` (без переустановки torch).

### 3.3. Откройте проект в PyCharm

`File → Open` → выбрать папку `clip_zero_shot_search`. PyCharm подхватит `.idea/`, но
интерпретатор нужно будет указать вручную при первом открытии:

`File → Settings → Project → Python Interpreter → Add Interpreter → Existing`
→ указать `.venv/bin/python` (или `.venv\Scripts\python.exe` на Windows), созданный на шаге 3.1.

После этого в выпадающем списке Run Configurations (верхняя панель) появятся четыре готовые
конфигурации:
- **1. build_index** — построение индекса
- **2. text_search** — пример поиска по тексту
- **3. image_search** — пример поиска по картинке
- **4. add_photos** — инкрементальное добавление новых изображений

Их параметры (`--query`, `--image_path` и т.д.) можно редактировать через
`Run → Edit Configurations...`.

## 5. Подготовка датасета

Скрипту `build` нужны:
1. Папка с изображениями (`.jpg`/`.jpeg`/`.png`), допускаются вложенные подпапки
   (например `images/val2017/*.jpg` — скрипт ищет рекурсивно).
2. CSV-файл с колонками `image_id`, `caption_en` (опционально `caption_ru` — скриптом
   не используется, поиск идёт по английским подписям).
   Один `image_id` может встречаться в нескольких строках (в MS COCO — 5 подписей на картинку).

**Формат имён файлов**: скрипт понимает стандартный для MS COCO формат с ведущими нулями
до 12 цифр (`image_id=139` → `000000000139.jpg`), а также имена файлов без паддинга
(`139.jpg`) как запасной вариант.

### Вариант А — скачать официальный MS COCO val2017

Linux / macOS / Colab:

Windows (PowerShell) — скрипт с докачкой и автоповтором при обрыве соединения:

.\.venv\Scripts\python.exe scripts\convert_coco_captions.py `

или скриптом с той же логикой, что в `scripts/convert_coco_captions.py`.

## 6. Использование (после установки и подготовки данных)

```bash
# Построение индекса (один раз; пересоздавать нужно только если датасет поменялся)
python src/clip_zero_shot_search.py build \
    --images_dir data/images \
    --captions_csv data/captions.csv \
    --index_dir index

# Поиск изображений по английскому тексту
python src/clip_zero_shot_search.py text \
    --index_dir index \
    --query "a dog playing in the snow" \
    --top_k 5

# Поиск изображений по русскому тексту (с автопереводом)
python src/clip_zero_shot_search.py text \
    --index_dir index \
    --query "собака играет в снегу" \
    --translate \
    --top_k 5

# Поиск похожих изображений и релевантных подписей по картинке-запросу
python src/clip_zero_shot_search.py image \
    --index_dir index \
    --image_path path/to/query.jpg \
    --top_k 5
```

После каждого поиска сохраняется коллаж с найденными картинками:
`index/search_result_text.png` или `index/search_result_image.png`
(отключается флагом `--no_plot`).

Метрика близости — косинусное сходство: эмбеддинги L2-нормализуются, а индекс FAISS —
`IndexFlatIP` (скалярное произведение), поэтому выводимый `score` лежит в диапазоне
примерно [-1, 1], где больше — ближе.

### 6.1. Добавление новых изображений в уже существующий индекс

Команда `add` добавляет новые фото в индекс, не пересчитывая эмбеддинги для уже
проиндексированных изображений — важно как при регулярном пополнении датасета, так и как
основа для Telegram-бота (раздел 9), где пользователь может прислать своё фото.

**Режим 1 — без CSV (типичный сценарий "пользователь прислал произвольное фото"):**
`image_id` берётся из имени файла без расширения. Подходит для фото без подписей —
они попадут в `images.index` (доступны для поиска "похожие картинки" и "картинка по тексту"),
но не появятся в результатах "картинка → подписи", так как подписи для них не заданы.

```bash
python src/clip_zero_shot_search.py add \
    --images_dir data/new_photos \
    --index_dir index
```

**Режим 2 — с CSV (пополнение датасета новыми размеченными изображениями):**
работает как `build`, но добавляет только новые записи поверх существующего индекса,
обновляя и `images.index`, и `captions.index`.

```bash
python src/clip_zero_shot_search.py add \
    --images_dir data/new_photos \
    --captions_csv data/new_captions.csv \
    --index_dir index
```

Повторный запуск `add` с теми же файлами безопасен — уже проиндексированные `image_id`
(с учётом ведущих нулей в имени файла, например `000000000139.jpg` и `139` считаются
одним и тем же id) автоматически пропускаются, дубликаты не создаются.

### 6.2. Оценка качества: Recall@1/5/10 и время ответа

```bash
# метрики по всему индексу (несколько секунд, CLIP не загружается)
python scripts/eval_recall.py --index_dir index

# + замер времени ответа на запрос и сохранение отчёта
python scripts/eval_recall.py --index_dir index --latency --output index/recall.json

# быстрая проверка на случайной подвыборке подписей
python scripts/eval_recall.py --index_dir index --limit 2000
```

Скрипт не пересчитывает эмбеддинги — он берёт готовые векторы прямо из `images.index`
и `captions.index` (`reconstruct_n`), поэтому работает за секунды. Модель CLIP грузится
только при флаге `--latency`. Изображения без подписей (например, присланные боту фото)
не используются как запросы, но остаются в индексе как дистракторы.

Считаются обе стандартные для MS COCO метрики:
- **text → image** — по подписи ищем изображение, правильный ответ один;
- **image → text** — по картинке ищем подписи, правильным считается попадание любой
  из пяти подписей этого изображения.

**Результат на текущем индексе** (val2017: 5000 изображений / 25 014 подписей, CPU):

| Направление | Recall@1 | Recall@5 | Recall@10 |
|---|---|---|---|
| text → image (25 014 запросов) | 30,41 % | 54,83 % | 66,25 % |
| image → text (5000 запросов)   | 49,96 % | 74,86 % | 83,16 % |

Среднее время текстового запроса — **~49 мс** (энкодер CLIP ~48 мс + поиск FAISS ~1 мс),
p95 ≈ 86 мс, всё на CPU. Требование ТЗ (< 1 с) выполняется с большим запасом.

Полученные значения совпадают с опубликованными для zero-shot **CLIP ViT-B/32** на COCO
5k test (i2t R@1 ≈ 50,1 %, t2i R@1 ≈ 30,4 %), что подтверждает корректность пайплайна.
Ориентир из ТЗ (R@1 ≈ 58 %) относится к более крупной модели CLIP (ViT-L/14@336) —
догнать его можно либо сменой чекпоинта на `openai/clip-vit-large-patch14`, либо
fine-tuning'ом (см. раздел 8).

## 7. Известные проблемы и их решения (из истории разработки)

Эти проблемы уже исправлены в текущей версии `src/clip_zero_shot_search.py`, но фиксирую их
здесь на случай похожих ошибок в будущем (например, при обновлении зависимостей):

| Проблема | Причина | Решение (уже в коде) |
|---|---|---|
| `AttributeError: module 'httpx' has no attribute 'TimeoutException'` | `googletrans==4.0.0rc1` требует старый `httpx==0.13.3`, конфликтует с `huggingface_hub` | Заменили перевод на `deep-translator` (не пинит старый httpx). **Не устанавливайте `googletrans`.** |
| Перевод возвращал тот же русский текст ("рандомные картинки" в результатах) | Неудачный перевод кэшировался как fallback, и следующие вызовы читали испорченный кэш | Функция `translate_ru_to_en` больше не кэширует неудачные попытки |
| `ValueError: need at least one array to concatenate` при `build` | `image_id_to_filename` не находила файлы — либо неверный формат имени, либо изображения лежат во вложенной подпапке (`images/val2017/*.jpg`) | Поиск теперь рекурсивный (по всем подпапкам) и понимает zero-padded формат имён MS COCO |
| `AttributeError: 'BaseModelOutputWithPooling' object has no attribute 'cpu'` | В `transformers >= 5.0` изменилось поведение `get_image_features`/`get_text_features` — возвращают объект, а не тензор напрямую | Добавлена функция `extract_features_tensor`, совместимая с обеими версиями API |
| Долгий поиск файлов на большом датасете | `rglob` вызывался заново на каждый `image_id` | Дерево папок теперь сканируется один раз (`build_filename_index`), дальше поиск по словарю |
| Команда `add` добавляла дубликаты, если новый файл назван с ведущими нулями (`000000000139.jpg`), а в индексе тот же снимок хранится как `image_id="139"` (без паддинга, как приходит из CSV) | Сравнение id "как есть" не учитывало разный формат представления одного и того же числового id | Добавлена функция `normalize_id` — числовые id сравниваются без ведущих нулей независимо от формата исходного имени файла |
| Бот в Docker не находил файлы картинок по путям из индекса | Индекс строился на Windows, в `images_meta.json` пути записаны с `\` и относительно корня проекта | Разделители приводятся к `/`, путь резолвится относительно корня проекта. В новых базах проблемы нет вовсе: в метаданных хранится только имя файла, а папка базы известна |

Если после обновления зависимостей `pip install -r requirements.txt` появляются новые
конфликты — самый надёжный первый шаг: создать чистое виртуальное окружение
(`python -m venv .venv` заново) и установить зависимости с нуля, не поверх старого окружения.

## 8. Возможные следующие шаги

- **Более крупный чекпоинт CLIP** (`openai/clip-vit-large-patch14`) — самый дешёвый способ
  подтянуть Recall@1 от текущих ~30 % (text → image) к ориентиру ТЗ ~58 %; цена — размер
  модели и время инференса на CPU.
- **Fine-tuning CLIP** на своём подмножестве COCO (contrastive/InfoNCE loss) для повышения
  Recall@k выше zero-shot базового уровня; альтернатива — переход на BLIP-2.
- Замена `IndexFlatIP` в FAISS на приближённый индекс (`IndexIVFFlat`, `IndexHNSWFlat`) при
  росте датасета выше ~100k изображений — точный поиск станет медленным.
- Поддержка `caption_ru` в индексе (сейчас колонка допускается во входном CSV, но не
  индексируется — поиск идёт только по английским подписям).
- Автоматические подписи к пользовательским фотографиям (BLIP) — тогда поиск
  «фото → подпись» заработал бы не только в демо-базе COCO.
- Проверка членства в чате через `getChatMember` с обновлением по времени: сейчас участник
  теряет доступ к базе чата, только когда бот об этом узнает.

## 9. Веб-интерфейс, Telegram-бот и Docker

### 9.0. Быстрый запуск

```bash
cp .env.example .env            # Windows: copy .env.example .env
docker compose up -d --build    # web + ui + bot
```

Интерфейс откроется на `http://localhost:8080` (порт задаётся `WEB_PORT`).
Зарегистрируйтесь логином и паролем — и можно загружать архив с фотографиями.

Для бота дополнительно нужны в `.env`:

```
TELEGRAM_BOT_TOKEN=...          # от @BotFather
SERVICE_TOKEN=<длинная случайная строка>
```

Без `SERVICE_TOKEN` ручек `/api/bot` не существует вовсе (404), и бот работать не будет.
Один и тот же токен читают оба сервиса — это внутренний пароль бота к API.

**Для групповых чатов обязательно**: у [@BotFather](https://t.me/BotFather) выполните
`/setprivacy` → **Disable**. По умолчанию Telegram показывает боту только команды,
и присланных в группу фотографий он просто не увидит.

Запуск без Docker (два процесса в разных терминалах):

```bash
pip install torch -r requirements-web.txt
uvicorn web.app:app --port 8000         # API, только один воркер

npm --prefix web-ui install && npm --prefix web-ui run dev   # фронтенд на :5173

python -m bot.bot                        # бот (нужны TELEGRAM_BOT_TOKEN и SERVICE_TOKEN)
```

### 9.1. Архитектура

```
  браузер ─────────────────┐
                           │        API (FastAPI)
                           ├──HTTP─▶ единственный процесс с CLIP
  Telegram ──▶ бот ────────┘        единственный писатель индекса
               без модели                    │
               250 МБ                       ├── SQLite: пользователи, базы, задачи
                                             └── data/users/<user>/databases/<db>/
                                                 индекс, оригиналы, превью
```

Модель загружается **один раз на процесс** и живёт резидентно (`core.model.ModelHolder`).
Отсюда три следствия, определяющие всю компоновку:

1. **API запускается строго с одним воркером.** Несколько воркеров — это несколько копий
   весов по 600 МБ и одновременная запись в один FAISS-индекс.
2. **Бот не считает эмбеддинги сам.** Он тонкий HTTP-клиент (`bot/client.py`): отдельный
   процесс с моделью означал бы вторую копию весов и второго писателя в те же файлы.
   Заодно Telegram и так допускает только одного потребителя `getUpdates` на токен.
3. **Логика не раздваивается.** И обработчики сайта, и ручки бота вызывают один
   сервисный слой (`web/services.py`), поэтому квоты, очередь задач и запрет на изменение
   демо-базы действуют одинаково в чате и в браузере.

Ранее в этом README (раздел 9.1 прошлой редакции) вариант с разделением на сервисы был
описан как шаг на будущее — он и реализован, но с уточнением: наружу вынесен **бот**,
а инференс остался внутри API, потому что выносить его отдельно означало бы гонять
эмбеддинги по сети без выигрыша — индекс всё равно пишет только API.

Тяжёлые операции (индексация архива на тысячи фото) уходят в очередь на одном рабочем
потоке (`web/jobs.py`) и отдают прогресс: на CPU это десятки минут, столько ни один
браузер соединение не держит.

### 9.2. Что умеет веб-интерфейс

- **Свои базы у каждого пользователя**: создать пустую или загрузить zip с фотографиями.
- **Поиск по описанию** (русский переводится автоматически) и **по картинке-образцу**.
- **Галерея** с постраничной подгрузкой и удалением снимков.
- **Выгрузка всей базы одним zip** — потоком, без сборки временного файла на диске.
- **Объём базы** в снимках и мегабайтах виден и в списке баз, и в шапке рабочей области.
- **Демо-база MS COCO** только для чтения: единственное место, где работает поиск
  «фото → подпись», потому что у пользовательских баз подписей нет.

Вход — логин и пароль, дополнительно можно привязать Telegram (`TELEGRAM_AUTH_ENABLED=1`,
нужен публичный HTTPS-домен, привязанный к боту через `/setdomain`). Восстановления пароля
письмом нет: SMTP не подключён. Запасные пути — вход через привязанный Telegram и команда
`python -m web.admin reset-password --login <логин> --password <новый>`.

### 9.3. Фронтенд

React + Vite, собирается в статику и отдаётся nginx. В разработке `npm run dev` поднимает
сервер на :5173 и проксирует `/api` на :8000 — так запросы остаются одноисточниковыми,
и cookie сессии (HttpOnly, SameSite=Lax) работают без послаблений.

### 9.4. Telegram-бот

У каждого чата **своя база**. Раньше индекс был общий, и присланное одним человеком фото
находилось по запросу любого другого; теперь этого нет.

| Команда | Что делает |
|---|---|
| `/start` | заводит базу этого чата; до неё бот ничего не сохраняет |
| фото | попадает в базу того чата, где отправлено |
| `/find <запрос>` | поиск; в личке можно писать без команды |
| `/stats` | сколько снимков и мегабайт накопилось |

Индексация в группе включается только командой `/start`: складывать чужие снимки на сервер,
никого не спросив, нельзя. В группе бот не отвечает на каждое фото, иначе превратился бы в спам.

Личный чат с ботом даёт ту же базу, что видна на сайте после входа через Telegram, — аккаунт
общий (таблица `identities`). Базу группового чата видят на сайте его участники: бот отмечает
их, когда видит сообщение.

### 9.5. Docker

| Образ | Что внутри | Размер |
|---|---|---|
| `Dockerfile.web` | FastAPI, CLIP (веса запекаются в образ), faiss | ~2 ГБ |
| `Dockerfile.ui` | сборка Vite, затем nginx со статикой | ~50 МБ |
| `Dockerfile.bot` | aiogram и httpx, без torch | 250 МБ |

Веса CLIP кэшируются в образ на этапе сборки, поэтому контейнер стартует офлайн
(`HF_HUB_OFFLINE=1`). Данные монтируются как volume и переживают пересборку. Порт API
наружу не публикуется — снаружи доступен только nginx.

Если сборка идёт из сети, где `pypi.org`/`huggingface.co` недоступны, задайте в `.env`
`BUILD_HTTP_PROXY`/`BUILD_HTTPS_PROXY` — они прокидываются в build-args и сбрасываются
на рантайме.

**За обратным прокси обязательно `TRUST_PROXY=1`** (в `docker-compose.yml` уже задано).
Ограничение частоты входа считает попытки по IP, а за nginx все запросы приходят с одного
адреса: без этого флага пятая неудачная попытка любого пользователя закрыла бы вход всем
остальным. Приложение берёт адрес из `X-Real-IP`, который nginx проставляет безусловно.

TLS не терминируется: поставьте перед nginx certbot/traefik либо пробросьте 443 с хоста.
Как только `PUBLIC_URL` начинается с `https://`, приложение само включает флаг `Secure`
у cookie сессии.

### 9.6. Тесты

```bash
pip install -r requirements-dev.txt
pytest tests -q                  # 162 теста, около минуты
```

Веса CLIP при этом не загружаются: модель подменяется детерминированной заглушкой
(`tests/conftest.py`). Проверяется не качество поиска — за него отвечает
`scripts/eval_recall.py` — а границы: приём zip (zip-slip, zip-бомбы, symlink-члены),
изоляция чужих баз, согласованность индекса и метаданных после удаления, права доступа
к базам чатов и демо-базе.

## 10. Соответствие ТЗ

Сводка по требованиям ТЗ «Интерфейс для Zero-Shot поиска изображений и текстовых описаний».

| Требование ТЗ | Статус | Комментарий |
|---|---|---|
| Zero-shot поиск без дообучения | ✅ | Готовые веса `openai/clip-vit-base-patch32` |
| Модель BLIP или CLIP | ✅ | Выбран CLIP (ТЗ допускает оба варианта) |
| Общее векторное пространство, косинусное расстояние | ✅ | L2-нормализация + FAISS `IndexFlatIP` |
| Векторная БД (FAISS) | ✅ | `images.index`, `captions.index` |
| Поиск изображений по тексту | ✅ | веб-интерфейс, `/find` в боте, команда `text` в CLI |
| Поиск похожих изображений по картинке | ✅ | перетаскивание снимка в веб-интерфейсе, команда `image` в CLI |
| Загрузка пользователем своих изображений | ✅ | zip-архив или отдельные файлы в вебе, фото в чат боту, `add` в CLI — везде инкрементально |
| Перевод запроса RU→EN через API | ✅ | `deep-translator` (Google), автоопределение кириллицы |
| Кэширование переводов | ✅ | `index/translate_cache.json` |
| Датасет MS COCO с подписями | ⚠️ | Используется val2017: 5000 изображений / 25 014 подписей. ТЗ в разных пунктах называет 1 тыс. изображений (п. 1.2) и 10 тыс. изображений / 50 тыс. подписей («Описание данных») — взято подмножество большего размера, чем минимально требуется |
| CSV с колонками `image_id, caption_en, caption_ru` | ⚠️ | Генерируется только `image_id, caption_en`; `caption_ru` допускается на входе, но не индексируется. Русские подписи в исходных аннотациях COCO отсутствуют |
| Отображение результатов с превью и score | ✅ | Сетка превью с полосой близости и косинусом в вебе, альбом в боте, коллаж PNG в CLI |
| Веб-интерфейс (Streamlit / FastAPI) | ✅ | FastAPI + React (раздел 9). У каждого пользователя своя база: загрузка архива, поиск текстом и картинкой, галерея, удаление, выгрузка zip, объём базы. Telegram-бот сохранён как второй интерфейс |
| Фильтрация по языку в интерфейсе | ⚠️ | Язык определяется автоматически по наличию кириллицы; переключателя нет, но интерфейс показывает строку, реально ушедшую в модель после перевода |
| Метрика Recall@1/5/10 | ⚠️ | Реализована (`scripts/eval_recall.py`, раздел 6.2). Получено: t2i 30,4 / 54,8 / 66,3 %, i2t 50,0 / 74,9 / 83,2 % — это ровно уровень zero-shot CLIP **ViT-B/32**; ориентир ТЗ (58 % R@1) соответствует более крупной ViT-L/14 |
| Среднее время ответа < 1 с (CPU + FAISS) | ✅ | Измерено: ~49 мс на запрос (энкодер ~48 мс + FAISS ~1 мс), p95 ≈ 86 мс на CPU |
| Примеры из мультиязычного WIT | ❌ | Не добавлены (в ТЗ — опционально) |
| Соблюдение лицензий данных и модели | ✅ | См. раздел 11 |

**Главные расхождения, о которых стоит знать:** в индексе нет русских подписей (их нет
в исходных аннотациях COCO), нет ручного переключателя языка, а Recall@1 ниже названного
в ТЗ ориентира — потому что ориентир взят для более крупной модели CLIP, чем используемая
ViT-B/32.

**Сверх ТЗ сделано:** вход по логину и паролю с привязкой Telegram, базы у каждого
пользователя и у каждого чата, фоновая индексация с прогрессом, выгрузка базы одним
архивом, квоты и 162 автоматических теста.
